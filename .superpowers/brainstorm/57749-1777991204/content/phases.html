Recommended 2-Phase Split

Phase 1 = everything needed to run the platform. Phase 2 = intelligence, revenue, and advanced ops.

Phase 1 — Platform Operations
Build this first · 7 domains · ship a working portal
Auth & Sessions API exists
Login, token refresh, session list & revoke
Tenant Lifecycle API exists
List, filter, search, view, suspend, reactivate, impersonate, upgrade plan
Tenant Onboarding UI gaps
Provision trial, schema setup, invite admin team, onboarding checklist
Plans & Licences API exists
View/create/edit plans, licence status, seat overrides, trial extension
Feature Flags API exists
Enable/disable per tenant, global defaults, plan-to-flag mapping
Audit & Compliance UI gaps
Cross-tenant log search, filter by domain/action/actor, export reports
Platform Config UI gaps
Update PAYE/NSSF/SHIF/Housing Levy rates, public holidays, country support
Phase 2 — Intelligence & Advanced Ops
Build after Phase 1 ships · 7 domains · mostly backend work
System Health & Monitoring
Service status aggregator, queue health, Redis, circuit breakers, dead-letter replay
Security & Incidents
Force logout, lock accounts, block IPs, failed login history, incident response
Billing & Revenue
MRR/ARR dashboard, invoices, payment history, gateway config, refunds
Support & Ops
Active users, engagement score, broadcast messages, GDPR export/purge
Communications
In-app messages, email campaigns, broadcast scheduling, email templates
Data Migration
Competitor import, schema mapping, preview, run & rollback
Backup & DR
Manual backups, restore from backup, archive cold tenants, DR testing

Why this split?

Phase 1 covers everything you need to onboard a customer, manage their lifecycle, control their plan and features, and stay audit-compliant — with most backend APIs already in place. Phase 2 is mostly net-new backend work (billing, monitoring aggregation, security event tracking) — ship Phase 1 first and have a usable portal while Phase 2 is built.
