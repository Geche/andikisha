## 2. Consumer Analysis

Every place in the codebase that touches a `PaySlip` entity or its data:

### `PayrollMapper` (`payroll-service`)
Uses MapStruct. The `toResponse(PaySlip slip)` mapping uses only the raw `BigDecimal` getters generated by Lombok's `@Getter`. MapStruct resolves field mappings by name match — it calls `slip.getGrossPay()`, `slip.getNetPay()`, etc. The `*AsMoney()` accessors are invisible to MapStruct and unused. `PaySlipResponse` carries raw `BigDecimal` fields to the REST API.

### `PayrollRun.finishCalculation()` (`payroll-service`)
Aggregates totals across all slips via `sumField(paySlips, PaySlip::getGrossPay)`, which calls raw `BigDecimal` getters through a `Function<PaySlip, BigDecimal>` reference. Uses `BigDecimal::add` for accumulation. Does not use and would not benefit from `*AsMoney()` accessors (see Section 3).

### `PaymentService` (`integration-hub-service`)
Never receives a `PaySlip` entity. It accepts raw parameters — `employeeId`, `amount` (`BigDecimal`), `currency` (`String`) — when creating a `PaymentTransaction`. The `PaySlip` entity boundary stops at the payroll-service.

### `document-service` (`PayrollEventListener` + `PayrollGrpcClient`)
The document-service never touches the `PaySlip` JPA entity. When a `PayrollProcessedEvent` fires, it calls the payroll-service gRPC endpoint (`getPaySlips`), which returns `PaySlipDetail` proto messages — not `PaySlip` Java objects. The proto fields (`gross_pay`, `net_pay`, etc.) are typed as `double` in `payroll.proto`. The listener converts them to `BigDecimal.valueOf(slip.getGrossPay())`. The `*AsMoney()` accessors are unreachable from this path.

### `analytics-service` (Phase 4)
The service is an empty shell — only an `@SpringBootApplication` main class exists. There is no implementation. Per the technical documentation, analytics will be CQRS read-side, consuming domain events via RabbitMQ. It will never call methods on a `PaySlip` JPA entity from a different service's database.

## 3. PayrollRun Aggregation

`PayrollRun.finishCalculation()` computes nine aggregate totals from the slip list:

```java
this.totalGross = sumField(paySlips, PaySlip::getGrossPay);
this.totalNet   = sumField(paySlips, PaySlip::getNetPay);
// ... seven more
```

```java
private BigDecimal sumField(List<PaySlip> slips, Function<PaySlip, BigDecimal> getter) {
    return slips.stream()
            .map(getter)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
}
```

This is raw `BigDecimal.add()` with no currency check. Using `Money` arithmetic instead (e.g. `Money.add()`) would enforce that all slips carry the same currency before summing, catching a theoretical cross-currency bug at runtime.

**However:** `PayrollRun.create()` hardcodes `run.currency = "KES"` and all slip currencies come from `salary.getCurrency().isBlank() ? "KES" : salary.getCurrency()`. A multi-currency run would require a deliberate product decision and a schema change to `PayrollRun`. The risk of silent cross-currency summation is theoretical, not operational.

**Judgment:** Money arithmetic in `finishCalculation()` would make the code *stricter*, not *clearer*. The benefit is marginal given the current single-currency constraint. If multi-currency support is ever introduced, the aggregation logic will need to be redesigned regardless — the existing `BigDecimal.add()` approach could not simply be swapped for `Money.add()`, because you would need to handle mixed-currency slips with separate totals per currency, not a single sum.

## 4. Planned Consumers

### TODOs and FIXMEs
No `TODO`, `FIXME`, or comment in any `payroll-service` Java file references slip aggregation, Money usage, reporting, analytics, or multi-currency handling.

### Analytics service
The product planning document describes a `PayrollAnalyticsView` read model for "payroll cost reporting by department, location, and cost centre." The technical architecture states analytics-service is "CQRS read side, never reads from other services' operational databases." It will be driven by events (`PayrollApprovedEvent`, `PayrollCalculatedEvent`), not by calling JPA entity methods. The `*AsMoney()` accessors would not be reachable from analytics.

### Multi-currency feature flag
The technical documentation notes a per-tenant `multi_currency` feature toggle exists as a roadmap item. There is no implementation and no schema support for it. If and when it is implemented, `PaySlip` would need a more substantial redesign (e.g. one currency per slip, aggregation in reporting layer) rather than simply calling `grossPayAsMoney()`.

### PDF payslip generation (`document-service`)
The `PayslipGenerator` already receives amounts as `Map<String, BigDecimal>` with separate gross and net `BigDecimal` values. It does not use `Money`. Changing it to use `Money` would require updating the generator API and the gRPC proto transport, neither of which has been planned.

**Conclusion:** No concrete planned consumer within the next phase of work (Phase 4 analytics) or any other documented feature can be identified that would call these accessors.

## 7. Comparison to `SalaryStructure`

`SalaryStructure` in `employee-service` uses full `@Embedded Money` per field — each monetary field is stored as two columns (`amount` + `currency`), and the class exposes amounts only as `Money` values. It provides a computed `grossPay()` method that uses `Money.add()` chaining to enforce currency consistency at the domain level.

This is the opposite pattern from `PaySlip`. `SalaryStructure` is a *mutable, live domain object* where currency arithmetic is an active concern — adding a USD housing allowance to a KES basic salary at the domain level would be a bug. The `Money` type catches that.

`PaySlip` is a *calculated snapshot*. By the time a payslip is written, all amounts have been computed from a single salary currency in `PayrollService`. The snapshot is immutable after creation. Currency arithmetic on the snapshot is not a concern — there is nothing to add or subtract. The accessors add a wrapping layer with no purpose at the snapshot's natural consumption points.

`PaySlip` is not an outlier. It is a different entity class that warrants a different exposure pattern, and the CLAUDE.md convention now explicitly documents this distinction.

## 8. Recommendation

**Delete all nineteen accessors and the `PaySlipMoneyAccessorTest` class.**

### Reasoning

**Against keeping:**

1. Zero production callers — not a single line outside `PaySlip.java` itself calls any of the nineteen methods.
2. The test file (`PaySlipMoneyAccessorTest`) was written to test the accessors — it is not evidence of consumer demand, it is a consequence of the accessors' existence. Deleting the accessors makes the test file obsolete.
3. The document-service, the only cross-service consumer of payslip data, uses `PaySlipDetail` proto messages — it will never call a JPA entity method.
4. The analytics-service is a shell. When implemented, it will read events, not entity methods.
5. `PayrollRun.finishCalculation()` uses raw `BigDecimal` method references and would need a different redesign if multi-currency runs became a requirement — not a simple accessor swap.
6. Adding nineteen methods to a public API creates nineteen future maintenance obligations. If the `currency` field or any `BigDecimal` field changes type, all nineteen accessors need updating.

**Against the "keep for future use" argument:**

The task instruction that created these accessors said "Expose Money via accessor methods at the application boundary." But the application boundary for `PaySlip` is `PayrollMapper`, and the mapper uses raw `BigDecimal` — not `Money`. The DTO carries `BigDecimal` to the REST API. The event payloads carry raw values. The gRPC proto carries `double`. None of these boundaries use `Money`, and changing them would require coordinated changes across multiple layers and services.

**What to keep instead:**

The `helbAsMoney()` null-guard pattern is correct and the null-safety audit found no issues. If a future consumer genuinely needs `Money` wrapping (e.g., a new in-process reporting service that sums payslips with currency enforcement), the accessors can be added at that point with a concrete caller. Writing them speculatively and then maintaining them indefinitely is the wrong trade.

**Action items after this decision:**
- Delete lines 127–145 of `PaySlip.java` (the nineteen accessor methods and the block comment above them).
- Delete `services/payroll-service/src/test/java/com/andikisha/payroll/unit/PaySlipMoneyAccessorTest.java`.
- Remove the `import com.andikisha.common.domain.Money;` from `PaySlip.java` (line 3 after the edit).
- Run `./gradlew :services:payroll-service:build` to confirm clean compile.
- No migration, no DTO, no mapper changes required.
