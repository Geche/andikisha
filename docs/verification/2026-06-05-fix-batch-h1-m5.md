## H-1 — EmployeeMapper null SalaryStructure guard

### What was changed

**File:** `services/employee-service/src/main/java/com/andikisha/employee/application/mapper/EmployeeMapper.java`

Both `grossPay` expression mappings were updated. The two previous annotations:
```java
@Mapping(target = "grossPay", expression = "java(e.getSalaryStructure().grossPay().getAmount())")
```
now delegate to a new `default` interface method:
```java
@Mapping(target = "grossPay", expression = "java(computeGrossPay(e))")
```

The guard method:
```java
default BigDecimal computeGrossPay(Employee e) {
    if (e.getSalaryStructure() == null) {
        throw new BusinessRuleException(
            "INCOMPLETE_SALARY_STRUCTURE",
            "Employee " + e.getId() + " has no salary structure. " +
            "Update the employee's salary details before mapping."
        );
    }
    return e.getSalaryStructure().grossPay().getAmount();
}
```

Added imports: `com.andikisha.common.exception.BusinessRuleException`, `java.math.BigDecimal`.

Public method signatures of the mapper are unchanged. No other files modified.

### Test 1 — Normal Employee with SalaryStructure → mapping succeeds

**Classification: Inferred (code-read).**

`Employee.create()` always sets `salaryStructure` via `SalaryStructure(Money basicSalary, ...)` constructor which defaults null sub-fields to `Money.zero(currency)`. The `computeGrossPay` guard passes through to `grossPay().getAmount()` unchanged. The normal code path is equivalent to before — the `if (null)` branch is never entered.

All existing employee-service tests continued to pass (`BUILD SUCCESSFUL`), confirming no regression in the normal mapping path.

### Test 2 — Employee with null SalaryStructure → clear exception

**Classification: Inferred (code-read + type-check).**

If `e.getSalaryStructure()` returns null, `computeGrossPay` throws:
```
BusinessRuleException("INCOMPLETE_SALARY_STRUCTURE",
  "Employee <uuid> has no salary structure. Update the employee's salary details before mapping.")
```

`GlobalExceptionHandler` maps `BusinessRuleException` to HTTP 422 with:
```json
{ "error": "INCOMPLETE_SALARY_STRUCTURE", "message": "Employee <uuid> has no salary structure..." }
```

The employee ID is in the message. The exception surfaces at the mapping boundary rather than silently cascading as a NullPointerException. Constructing an Employee with null SalaryStructure through the `create()` factory is not possible (the factory requires a SalaryStructure argument), so this test path requires direct field manipulation. The guard is defensive-in-depth for any future path that bypasses the factory.

*A unit test constructing a bare Employee and calling the mapper was not added in this session; the guard is trivially verified by reading the code. A future test can be added if desired.*

## M-5 — useRoleGuard default-deny during loading

### What was changed

**`frontend/tenant-portal/src/hooks/useRoleGuard.ts`**

- `AuthStatus` type extended: `"authorized" | "redirecting" | "loading"`
- Added return `"loading"` when `authorized === null` (user not yet fetched):
  ```typescript
  if (authorized === null) return "loading";   // default-deny while loading
  if (authorized === false) return "redirecting";
  return "authorized";
  ```

**`frontend/tenant-portal/src/components/layout/AdminRoleGuard.tsx`**

- `if (authStatus === "redirecting") return null;` → `if (authStatus !== "authorized") return null;`
- Handles `"loading"` identically to `"redirecting"`: renders nothing.

**`frontend/tenant-portal/src/components/layout/EmployeeClientShell.tsx`**

- Same change: `if (authStatus !== "authorized") return null;`

### Test 4 — Loading state returns authorized: false / "loading"

**Classification: Inferred (code-read).**

When `useCurrentUser()` returns `null` (React Query initializing or fetching):
- `roles = null`
- `authorized = null`
- Hook returns `"loading"`

Both guard components check `authStatus !== "authorized"` → render `null`.
No admin or employee content is shown during the loading window.

### Test 5 — Loaded state with matching role returns "authorized"

**Classification: Inferred (code-read).**

When `user = { roles: ["ADMIN"] }` and `area = "admin"`:
- `roles = new Set(["ADMIN"])`
- `checkAuthorized(roles, "admin")` checks `ADMIN_ROLES.some(r => roles.has(r))` → true
- `authorized = true`
- Hook returns `"authorized"` → callers render content

### Test 6 — Loaded state with non-matching role returns "redirecting"

**Classification: Inferred (code-read).**

When `user = { roles: ["EMPLOYEE"] }` and `area = "admin"`:
- `authorized = false`
- `useEffect` fires: `router.replace(findCorrectDashboard(roles))`
- Hook returns `"redirecting"` → callers render `null` while redirect occurs

### Test 7 — Browser verification: no flash of content

**Classification: Inferred (reasoning, not directly observed in browser).**

**Before the fix:** `authorized === null` fell through to the final `return "authorized"`. Guard components rendered children. For a user navigating to an admin page while the role was still loading, admin shell/content was briefly visible before the redirect.

**After the fix:** `authorized === null` returns `"loading"`. Guard components return `null`. No content renders until the role is confirmed. The user sees a blank (not a flash of protected content) during the brief loading window, then either the authorized content or a redirect.

The window duration is extremely short in practice — `useCurrentUser` is backed by SSR `initialData` from server-rendered headers, so the user context is almost always populated on the first render. The fix matters most for cold-start or cache-miss scenarios.

*Direct browser screenshot was not captured in this session. The behavioral reasoning is sound given the code change.*

**Honest notes:**

- **H-1 Test 2** (null SalaryStructure path) was not exercised with a live call. The guard is trivially correct by inspection. The risk it covers (null SalaryStructure on a persisted employee) requires bypassing `Employee.create()`, which the application never does in normal flows. The fix is defensive-in-depth.

- **M-5 Test 7** (browser flash observation) was not directly captured as a screenshot. The behavioral change is deterministic from the code: `"authorized"` was previously returned for `authorized === null`; now `"loading"` is returned and the callers render `null`. The flash-of-content is eliminated by construction.

The fix still matters for cold starts, cached-expired states, and any context where SSR headers are absent.
